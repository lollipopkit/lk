# LK Macros Roadmap

LK macros follow a Rust-shaped surface syntax with LK semantics. The goal is a full macro ecosystem that feels familiar to Rust users while staying compatible with LK's parser, module system, type checker, VM, WASM, LSP, and future native backends.

## Implementation Phases

### 1. Declarative Macro Foundation

- Keep the macro implementation split into focused modules as it grows so each file stays under the 1500-line limit.
- Replace the remaining scanner fallback paths with dedicated fragment parsers where practical, and keep extending follow-set diagnostics for nested matcher edge cases.
- Keep nested repetition coverage broad: arity checks, separator validation, optional repetition behavior, zero-width repetition rejection, duplicate-binding checks, matcher/template repetition-shape validation, empty capture shapes, trailing-separator rejection, and nested separator rendering are covered; add new parser-fragment edge cases as LK grammar grows.
- Preserve span mapping and token origin stacks from call-site tokens, template tokens, `$crate` anchors, and provider output so diagnostics point to the useful source location and explain the macro stack that produced it.
- Add negative tests for ambiguous matchers, unknown fragment kinds, unmatched rules, repetition arity mismatch, recursion limit, and hygiene collisions.

### 2. Macro Namespace and Modules

- Keep file and package macro imports aligned with LK `use`: `use { name as alias } from "file"` and `use { name as alias } from package` import compile-time macro bindings; `use "file"` / `use package` expose `namespace::name!`; `use * as ns from "file"` / `use * as ns from package` expose `ns::name!`. Do not mix runtime item imports and named macro imports in the same `use`; split them into separate statements.
- Keep growing the standard compile-time-only `macros` module beyond the current assertion/control-flow baseline while preserving LK package semantics rather than copying Rust attributes exactly.
- Keep explicit macro export controls strict: external file/package/std macro imports only see `export macro_rules!` definitions and `export { name as alias };` re-exports. Ordinary `macro_rules!` definitions stay usable in the defining file/module and through `$crate::helper!()` from exported macros, but remain private to external importers.
- Keep `$crate` anchors reliable for private helper macro calls and generated runtime item references. Same-file anchors rewrite to local item references; imported file/package anchors inject a generated runtime namespace import and lower `::` to LK runtime member access.
- Extend definition-module anchors to any future runtime reference forms that need more than namespace item/member access.
- Ensure runtime imports, package imports, REPL sessions, LSP workspace caches, and WASM all share the same macro resolver behavior.

### 3. Hygiene and Diagnostics

- Keep tracking whether each token originates from the call site, macro definition, `$crate` anchor, or process-backed provider output.
- Keep expanding hygienic freshening beyond the current local binding coverage (`let`/`const` destructuring, short declarations `name := value`, `for` simple bindings plus tuple/array/object destructuring and rest bindings, `if let`/`while let`, `match` arm patterns and guards, `select case value <- recv(...)` bindings, positional/named/default function parameters plus pipe closures and `fn(params) => expr/block` closure parameters, generated semantic name and item/member definition name preservation for member fields, map keys, struct literal fields, named argument keys, attribute tokens, `use` statement item/alias/source names, `::` path segments, `fn`/`struct`/`trait`/`type` definitions, struct declaration fields, generated type-reference preservation for local bindings, function positional/named/default parameter/return types including named function-type annotations, type aliases, and impl headers including complex function-type targets, plus lexical scope-limited renaming for generated block-local `let`/`const`, short declaration, for-body, if-let-body, while-let-body, match-arm, select case, and function-body and closure-body parameter ranges, initializer/value/iterable/default-expression exclusion boundaries, plus no-body signature parameter preservation) toward a real scope-aware hygiene model.
- Keep extending macro-origin diagnostics beyond the current parse/type/LSP type-diagnostic path. AST macro origins now appear in hover and document symbols, AST generated item origins carry structured source-map spans, external AST macro output records generated function/trait/impl member origins, top-level generated statement and control-flow origins including expression statements, `break`, and `continue`, generated expression category origins for literals, list/map/binary/unary/paren, optional access, conditionals, ranges, closures, expression blocks, named calls, struct literals, and template strings without treating those category labels as goto-definition targets, generated field-access and static index expression origins including chained forms such as `items.0.render`, generated call-callee reference origins, generated variable reference origins including match-arm guard references, generated assignment target reference origins, generated binding origins, generated match-arm origins, generated select case origins including `default`, generated semantic-name origins including map/object destructuring pattern keys, generated import origins, generated attribute origins, generated range-pattern reference origins, generated type reference origins including generated declarations in top-level items and nested statement/block bodies, and generated declaration labels such as `type Alias`, `struct Boxed`, `trait Reader`, `fn read`, plus struct declaration field origins; built-in derive generated member/expression origins expose `fn show` and `expr self.field`; same-file plus imported file/package macro calls can jump to their `macro_rules!` definition; and ordinary references can jump to `macro_rules!` plus manifest-backed proc/AST generated item definitions and generated members/field/index/call/ref/assign/binding/semantic-name/import/attribute/range-pattern/top-level-statement/control-flow/type/generated-declaration expressions plus generated impl item labels, including map/object destructuring pattern key origins, static-index chained call origins, generated select case origins, generated attribute/derive origins, and generated named function-type parameter origins and generated type-variable origins, through expanded token spans and AST generated origins. Next extend generated-symbol navigation to the remaining AST-generated reference categories.
- Keep the current expansion trace API and CLI command aligned with structured origin backtraces; `lk macro expand --origins` is the canonical token-level and AST item-level source-map dump.

### 4. Procedural Macro Protocol

- Keep the implemented versioned protocol data structures stable enough for early experimentation.
- Input already models macro kind, macro name, token stream, spans, current package/module identity, feature flags, and protocol version.
- Output already models token stream, diagnostics, optional notes, and deterministic dependency metadata; expansion APIs collect dependency metadata from derive, attribute, and function-like providers.
- Dependency metadata is fingerprinted using the provider-reported `path`/`digest` plus the resolved file state when the path is readable. Direct native execution stores a `.proc-macro-deps.json` sidecar beside each cached native executable and refuses stale cache hits. The LSP workspace cache stores the same fingerprint, invalidates preloaded analysis when a dependency file changes or appears after being missing, and maintains a reverse dependency graph so saving a proc-macro dependency file immediately evicts cached LK files that depend on it.
- Output tokens preserve provider-supplied spans through token expansion and post-parse AST macro expansion; missing output spans fall back to the macro call or attribute span. Context-sensitive provider token kinds such as `Or` / `||` are preserved instead of re-lexed as standalone source fragments. Function-like provider output also carries token-level `proc_macro_output` origins.
- Keep the process host enforcing timeout, output size limit, protocol version checks, process error isolation, and deterministic failure messages.
- Keep `ProcMacroProviders` as the compiler-facing registry and `[macros.derive.NAME]`, `[macros.attribute.NAME]`, and `[macros.function_like.NAME]` as the manifest-facing provider schema.
- Keep proc macros outside the compiler process; do not use `unsafe` outside the LLVM boundary.

Manifest provider shape:

```toml
[macros]
trusted_dependencies = ["helper_macros"]

[macros.derive.MakeAnswer]
command = "./tools/derive-make-answer"
args = ["--json"]
timeout_ms = 5000
max_output_bytes = 1048576

[macros.attribute.route]
command = "./tools/route-attr"

[macros.function_like.sql]
command = "./tools/sql-macro"
```

Local providers are always available to the package that declares them. Providers
declared by dependencies are discovered only after the current package explicitly
lists the dependency in `[macros].trusted_dependencies`. Trusted dependency
function-like macros are registered under the dependency namespace
(`helper_macros::sql!()`); trusted dependency derive and attribute providers use
their declared names because current derive/attribute syntax is not path-shaped.
Local providers win name collisions.

### 5. Derive and Attribute Macros

- Keep the implemented attribute preservation layer transparent across parser, type checking, slot resolution, VM compilation, REPL, LSP, tree-sitter, and display.
- Keep the built-in `#[derive(Debug)]` / `#[derive(Show)]` struct expansion generating the internal `show(self) -> String` method that template strings and formatted output already use.
- Keep the built-in `#[cfg(...)]` item filter aligned with Rust-style predicates and wire package feature resolution into `ParseOptions::macro_features`.
- Keep attributes on impl methods routed through the same post-parse attribute macro expansion pass; method macros must expand to zero or one function method.
- Keep `#[derive(Name)]` accepting built-in names and registered external derive providers; unregistered external derives must report a provider diagnostic instead of failing during attribute parsing.
- Keep external derives working for struct, type-alias, and trait inputs, then extend them to additional type declaration forms and generated-span categories through the proc macro protocol.
- Keep registered `#[attr] item` macros transforming, replacing, or removing a single item through the proc macro protocol; unregistered attributes remain preserved wrappers.
- Keep `#[attr]` impl-method macros constrained to method-level output so generated impl bodies remain structurally valid.
- Update type checking and symbol collection so generated items participate normally in later compiler phases.

### 6. Ecosystem Tooling

- Keep improving `lk macro expand <file>` with optional token trace output, `--deps` procedural dependency output, `--origins` token/AST/generated-item source-map output, and AST derive/attribute expansion output as the canonical expansion inspection command.
- Keep improving LSP diagnostics, hover, document symbols, goto-definition, macro definitions, imported macro definitions, and macro calls for macro expansion failures and generated code.
- Keep macro dependency fingerprints wired into cache/watch users. Native direct-run cache, LSP workspace cache, dependency-save reverse invalidation, directory dependency invalidation for saved files under proc-macro dependency directories, and the shared `ProcMacroDependencyGraph` model in `core` are covered; future project-wide incremental compilation should use the same fingerprint and reverse-graph model instead of inventing a parallel dependency tracker.
- Keep tree-sitter, VSCode grammar, website specs, README, and examples in sync with each macro phase.
- Keep `lk pkg check` and `lk pkg publish --dry-run` as the local/Git and registry-manifest distribution gates. `lk pkg publish --dry-run` emits a deterministic `integrity.sha256` digest over the publish manifest payload; `lk pkg publish` verifies that digest and uploads the same manifest to `[registry].url/api/v1/packages` with `LK_REGISTRY_PUBLISH_TOKEN` / `LK_REGISTRY_TOKEN` / `LK_PUBLISH_TOKEN` and publish scope headers; `lk pkg fetch` resolves exact and semver-range registry version dependencies to Git `source` plus locked `rev`, validates optional registry-provided `publish_manifest` metadata before trusting a version, skips yanked entries during range selection, verifies optional `sha256:<hex>` package checksums, and stores the checksum in `Lk.lock`; `lk pkg yank` / `--undo` manage yanked versions through authenticated registry requests with yank scope headers and `LK_REGISTRY_YANK_TOKEN` support; `lk pkg index sync` caches `[registry].url/api/v1/index` into `$LK_HOME/registry/<registry-name-or-url-key>/index.json` with version/checksum/yank/macro-provider metadata and optional publish manifests; `lk pkg fetch --offline` and `lk pkg update [name] --offline` resolve registry dependencies from that local index without registry HTTP requests. Later grow the server side with immutable content storage and cryptographic signing over the same integrity payload.

## Acceptance Matrix

- `macro_rules!` examples equivalent to `vec!`, assertion macros, `matches!`, panic-family macros, and control-flow helpers execute through CLI, VM compilation, WASM, and imports.
- Cross-file exported macros work from package dependencies and workspace members.
- Macro expansion failures plus macro-generated parse/type diagnostic failures include rule names, call-site spans, token origins, and an expansion stack.
- Procedural macro crashes or timeouts never crash the LK compiler.
- Procedural macro dependency metadata invalidates direct native cache hits and LSP workspace cache entries when the referenced dependency files change, and LSP save handling invalidates cached LK files through a proc-macro dependency reverse graph.
- Package publishing validates macro distribution metadata, prints the publish manifest with `--dry-run` including a deterministic `integrity.sha256` digest, refuses to upload tampered manifests, uploads the scoped authenticated manifest JSON to registry mock endpoints, resolves exact plus semver-range registry version dependencies to checked, locked Git revisions, validates optional registry-supplied publish manifests, yanks/un-yanks registry versions with scoped auth headers, syncs the registry index cache, and resolves registry dependencies from that cache in offline mode without requiring real network services in tests.
- AST macro origins appear in LSP hover and document symbols so generated items, generated control-flow nodes, generated imports, and generated attributes remain discoverable from their source macro input.
- Same-file and imported file/package macro invocations support LSP goto-definition to the defining `macro_rules!` name, including named imports, namespace imports, aliases, and macro re-export aliases.
- Ordinary references to `macro_rules!`-generated item definitions support LSP goto-definition through expanded token spans.
- Ordinary references to manifest-backed proc/AST generated item definitions plus external/built-in generated members, field expressions, call-callee references, variable references, assignment target references, binding origins, semantic-name origins, import origins, attribute origins, range-pattern references, top-level generated statement and control-flow references, type references, generated declaration labels, and generated impl item labels support LSP goto-definition through expanded token spans and AST generated origins.
- Generated code is visible to type checking, LSP symbols, semantic tokens, and native/LLVM compilation.
